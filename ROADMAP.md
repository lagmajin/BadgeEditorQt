# BadgeEditorQt Roadmap

## 目的
BadgeEditorQt を、以下の条件を満たす実用的なバッジ編集アプリに育てる。

- 今ある機能を壊さない
- Designer と Layout を完全に分離する
- Qt に依存しない純データ層を増やす
- 画像読み込みと出力を安定させる
- 大量作成でも破綻しない操作性にする

## 設計方針

- UI は操作の入口だけを担当する
- データは Qt 非依存のモデルに寄せる
- レイアウト計算と画像処理はサービス層に閉じ込める
- Designer は編集用キャンバス、Layout は配置確認用キャンバスとして分ける
- 画像読み込みは OIIO を優先し、Qt の読み込みは補助扱いにする
- 仕様が増えても壊れにくいよう、まず境界を整理してから機能を足す

## 現在地

- Designer 画面は編集の中心として動作している
- Layout 画面は別ウィジェットとして分離済み
- 保存/読込は JSON ベースで動作している
- レイアウト自動配置は `LayoutEngine` にある
- 純データの一部は C++ モジュール化済み
- 画像読み込みは OIIO 経由に寄せ始めている
- 用語の整理は [GLOSSARY.md](./GLOSSARY.md) にまとめる
- サイズ、DPI、出力の基準は [SPEC.md](./SPEC.md) にまとめる

## フェーズ 1: 境界の整理

### 目標
- MainWindow を配線専用に近づける
- Designer と Layout を完全に別の責務にする
- 画面ごとの状態更新を明確にする

### 作業
- ドキュメントの元データを一本化する
- Designer 側の選択/編集ロジックを整理する
- Layout 側は「表示するだけ」に寄せる
- Inspector の更新経路を単純化する

### 完了条件
- Designer と Layout が互いの内部実装を知らない
- MainWindow が scene や item を直接いじらない
- 画面切り替え時に状態が破綻しない

## フェーズ 2: 純データ層の拡張

### 目標
- Qt 非依存のモデルを増やす
- そのままテストしやすい構造にする

### 作業
- `BadgeData` と `LayerData` を中心に据える
- `PaperConfig` を純データ側に寄せる
- レイアウト結果を純データで扱う
- JSON 変換は Qt 側アダプタに分離する

### 完了条件
- レイアウト計算を Qt なしで呼べる
- 保存データの構造が UI 変更に引きずられない
- 将来のユニットテストが書きやすい

## フェーズ 3: 画像処理の安定化

### 目標
- PNG/JPEG 読み込みを安定させる
- 画像処理の入口を統一する
- 入力画像を可能な範囲で sRGB に正規化する

### 作業
- OIIO を画像読み込みの中心にする
- サムネイル生成とデコードを分離する
- 読み込み失敗時のフォールバックを明確にする
- 色補正の反映経路を整理する
- 埋め込み色空間がある画像は表示用に sRGB へ変換する

### 完了条件
- PNG/JPEG 読み込みが Qt 依存だけに見えない
- 画像が壊れていてもアプリが落ちない
- 画像の状態管理がひとつの経路にまとまる
- 一般的な埋め込みプロファイル付き画像の表示色が大きく崩れない

## フェーズ 4: 編集体験の強化

### 目標
- 実際の編集作業で使いやすくする
- バッジ編集ソフトとしての完成度を上げる

### 作業
- Undo/Redo をコマンド化する
- 複数選択と整列を整える
- レイヤー順変更を安定させる
- 複製、削除、整列、スナップを追加する

### 完了条件
- 移動、サイズ変更、画像差し替えが履歴に残る
- 失敗しにくい編集フローになる
- 大量のバッジを扱っても操作感が崩れない

## フェーズ 5: 出力の完成

### 目標
- 印刷と納品に耐える出力を作る

### 作業
- PNG 出力を実装する
- PDF 出力を実装する
- レイアウト画面と出力結果を一致させる
- トンボ、余白、安全圏の扱いを統一する

### 完了条件
- 画面上の見え方と出力結果の差が小さい
- 一括出力が安定する
- 印刷前チェックがしやすい

## フェーズ 6: モジュール化の拡張

### 目標
- 純ロジックをさらに切り出しやすくする

### 作業
- 画像入出力モジュールを増やす
- レイアウト補助関数を module 化する
- Qt 変換アダプタを境界に固定する

### 完了条件
- Qt に触れないコードが増えている
- 純ロジックの再利用性が高い
- 将来の非Qt利用やテストがしやすい

## 今の優先順位

1. Designer と Layout の責務分離を崩さず進める
2. 保存モデルの一本化を進める
3. OIIO ベースの読み込みを安定させる
4. Undo/Redo を入れる
5. 出力を実装する

## しばらく保留でよいもの

- 高度なテンプレート共有
- クラウド同期
- 共同編集
- 外部プラグイン機構
- 複雑な非破壊エフェクト

## 判断基準

- 既存機能を壊さないことを最優先にする
- 画面追加より責務分離を優先する
- Qt 依存を増やす前に、純データに寄せられるか考える
- 一度に大改修しない
- 途中で戻せる小さな単位で進める
