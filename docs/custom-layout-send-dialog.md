# カスタムレイアウト送信ダイアログの詳細設計案

この文書は、通常のレイアウト送信とは別に、同一デザインのサイズ違いを同数で送るための「カスタムレイアウト送信」ダイアログのたたき台です。

## 目的

- 既存の通常送信を壊さない
- 特殊なレイアウトだけを別フローに分離する
- 同じデザインの `32mm` と `57mm` などを同数で扱いやすくする
- 将来の拡張を見越して、入力を一箇所に閉じ込める

## 基本方針

- 通常のレイアウト送信は今まで通りにする
- カスタムレイアウト送信だけ別ダイアログを開く
- ダイアログで作った設定は、送信用の一時データに変換してから Layout に渡す
- 保存データの構造は極力変えない

## 想定する入口

- メイン画面に `通常送信` と `カスタム送信` の2つを置く
- あるいは `送信` ボタンのメニューで分ける

最初はボタンを分けるより、メニュー分岐の方が既存UIを崩しにくい。

## ダイアログに入れる項目

### 必須

- 元デザインの対象
- サイズ一覧
- 各サイズの枚数

### あると便利

- サイズごとの並べ順
- サイズごとのページ分割
- 同一ページに混在させるかどうか
- 既定の余白や間隔

## 最小構成

最初の版では、次の3つだけで十分。

1. サイズ `mm`
2. 枚数
3. 配置順

例:

- `32mm / 20枚`
- `57mm / 20枚`

## UI 構成案

### 上部

- 説明文
- 今回の送信対象のデザイン名

### 中央

- サイズ設定の一覧
- `追加` ボタン
- `削除` ボタン

### 下部

- `通常送信`
- `プレビュー`
- `キャンセル`

## サイズ設定行の中身

1 行につき、次を持つ。

- サイズ `mm`
- 枚数
- 必要ならタグやメモ

将来の拡張を見越すなら、行ごとに次も持てる。

- 回転
- 余白
- 個別の配置ルール

ただし最初は入れない方が安全。

## 送信データの考え方

カスタム送信ダイアログは、最終的に次のような一時データを作る。

- 元デザインID
- サイズ
- 枚数
- 生成順

その一時データから、内部で複製された `BadgeItem` 群を作る。

## 実装の責務分離

### 既存フロー

- Designer の編集
- 通常の Layout 送信
- 今の自動配置

### 新フロー

- カスタム送信ダイアログ
- サイズごとの一時リスト生成
- サイズ別配置

この分離により、通常動作の影響を最小限にできる。

## 開発順

### 1. 設計だけ固める

- ダイアログに入れる項目を決める
- 通常送信との分岐点を決める

### 2. 一時データの形を決める

- `sizeMm`
- `count`
- `sourceBadge`

### 3. ダイアログを作る

- 一覧編集
- 追加
- 削除
- 簡易プレビュー

### 4. Layout への変換を作る

- サイズごとに複製
- 数量分を展開
- 既存レイアウト関数へ渡す

## 既存ソースを壊しにくいコツ

- `BadgeItem` の定義をすぐ変えない
- 保存形式に新しい必須項目を増やさない
- `Designer` の状態をそのまま Layout に持ち込まない
- まずは送信時の変換だけで完結させる

## 仕様決定が必要な点

- 同じデザインを複数サイズにする時、サイズごとにページを分けるか
- 32mm と 57mm を同一ページに混在させるか
- サイズ違いは同数固定にするか、個別枚数にするか
- 送信時に自動で並べ替えるか、順番を維持するか

## 変換ルール案

ダイアログで受けた設定を、Layout 用の一時リストへ変換する時のルールを決めておく。

### 1. 元デザインを複製する

- 元の `BadgeItem` をベースにする
- サイズだけ差し替える
- 見た目用の状態はできるだけそのまま使う

### 2. 枚数分だけ展開する

- `20枚` なら 20 件作る
- 1 件ごとに個別の `BadgeItem` として扱う
- 位置はまだ決めない

### 3. サイズごとにまとめる

- 32mm 群
- 57mm 群

この順番は、あとでユーザーが見返した時にわかりやすい。

### 4. レイアウトエンジンに渡す

- 既存の `LayoutEngine` の入力へ流す
- そこで紙面配置を行う

## 状態の持ち方

ダイアログ内部では、次のような一時状態を持つと扱いやすい。

- `sourceBadge`
- `sizeMm`
- `count`
- `groupName`
- `order`

最初は `sourceBadge` / `sizeMm` / `count` だけでも十分。

## 画面のイメージ

最低限は、表形式で足りる。

| サイズ | 枚数 | 備考 |
| --- | --- | --- |
| 32mm | 20 |  |
| 57mm | 20 |  |

操作としては次があるとよい。

- 行追加
- 行削除
- 上へ移動
- 下へ移動
- 合計枚数表示

## 合計枚数の扱い

- サイズごとの枚数は個別に持つ
- 全体の合計も表示する
- 印刷枚数の見通しを立てやすくする

## エラー条件

- サイズが空
- 枚数が 0 以下
- 同じサイズの重複
- 紙面に収まらない

最初は、重複と空値だけでも検出できれば十分。

## まず決めるもの

実装前に、最低限この3つを決めるとブレにくい。

1. サイズごとにページを分けるか
2. サイズごとの枚数は同数固定か
3. カスタム送信の入口をどこに置くか

## おすすめの初期方針

- 通常送信はそのまま
- カスタム送信だけ別ダイアログ
- サイズごとにページを分ける
- 各サイズは同数を基本とする
- 混在配置は後回し

## ひとこと

この分け方なら、今ある完成した部分をあまり触らずに、新しい使い方だけを追加できる。  
「普通は今まで通り、特殊な時だけ詳細設定」という設計にしておくと、運用も保守も楽になる。
